Abstract

Stock Metrics is a full-stack stock market analysis web application designed to provide investors, students, and finance learners with a single integrated platform for multi-market research, screening, portfolio observation, price alerts, learning resources, and AI-assisted decision support. The project was developed as a static-first, local-first system using Next.js, TypeScript, Tailwind CSS, Zustand, React Query, IndexedDB, and a set of lightweight route handlers that proxy and cache free public data sources. Its core design philosophy is that useful investment analysis software should remain accessible even when paid market APIs, premium financial terminals, and complex backend infrastructure are unavailable.

The application supports United States equities, Indian equities, and Indian mutual funds. It provides universal search across markets, quote and historical price retrieval, best-effort fundamentals, charting, news aggregation, document discovery, portfolio and watchlist management, Excel export, beginner and professional user modes, optional authentication adapters, rule-based AI analysis, natural-language screening, and a personalized agentic recommendation engine that evaluates household financial profile, risk, liquidity, and market scope before producing stock suggestions. The project is deliberately engineered to degrade gracefully: if live data is unavailable, the application can fall back to deterministic demo or cached data rather than failing completely.

This report documents the complete project from objective and architecture to algorithms, implementation modules, data sources, testing results, limitations, and future scope. The contents were prepared by inspecting the repository, reading the implemented source files, reviewing the active page and API structure, and validating the project through successful execution of the existing automated tests and production build.

Keywords: stock analysis, Next.js, TypeScript, financial dashboard, local-first app, personalized investment engine, screener, market data, AI heuristics, portfolio tracking.

Executive Summary

In practical terms, Stock Metrics attempts to answer a simple but important problem: many retail users need one clean workspace to study stocks, compare metrics, maintain personal watchlists, track small portfolios, learn investing concepts, and receive basic automated guidance, yet most available platforms either lock critical features behind subscriptions, specialize in only one geography, or assume that the user already understands financial language. This project addresses that gap by combining research tools, learning support, and explainable AI into a single responsive web application that can run with minimal setup.

The most distinctive technical contributions of the project are its free-data-first approach, modular adapter architecture, client-side persistence model, explainable heuristic AI layer, and the personalized agentic engine that combines user financial profile analysis with stock-level scoring. Rather than being a simple CRUD dashboard or a static finance site, the project demonstrates a thoughtful integration of frontend engineering, data ingestion, caching, domain modeling, browser persistence, optional external integrations, and test-backed business logic.

1. Introduction

The stock market is rich in information but fragmented in delivery. Investors frequently move between market quote pages, financial statement websites, screeners, portfolio trackers, news portals, learning resources, and notes applications to build a single view of a company or investment idea. This fragmentation increases the effort required for even basic analysis and creates a particularly difficult learning curve for beginners. At the same time, many advanced platforms depend on expensive APIs, premium subscriptions, or backend-heavy infrastructure that makes them harder to self-host, experiment with, or study as a student project.

Stock Metrics was designed as a response to these issues. It is a research-oriented finance web application that unifies stock discovery, multi-market analysis, charting, documents, news, notes, screening, learning resources, alerts, local portfolio tracking, and AI-based interpretation under one system. The project places strong emphasis on accessibility, graceful degradation, and educational value. It is not positioned as a brokerage or real-time trading terminal; instead, it is a technically thoughtful decision-support and learning platform for delayed market research and personal analysis workflows.

Architecturally, the system follows a browser-centric model. User-owned data such as watchlists, transactions, notes, custom screens, session state, and cached analysis artifacts are stored locally in the browser using IndexedDB, localStorage, and sessionStorage. External data is accessed through reusable adapters and minimal route handlers that proxy free public providers such as Yahoo Finance, SEC EDGAR, Google News RSS, MFAPI, AMFI, and ER-API. AI behavior defaults to deterministic heuristics so the application remains usable without a paid model provider, while optional integrations for Ollama, OpenAI-compatible endpoints, Firebase Authentication, and Gmail SMTP remain available as extensions.

2. Problem Statement, Existing System, and Proposed Solution

2.1 Problem Statement

The central problem addressed by this project is the absence of an accessible, student-friendly, low-cost platform that combines stock research, multi-market discovery, personalized analysis, portfolio observation, learning support, and AI-assisted insight in one coherent web application. Many investors either rely on multiple disconnected services or are forced into premium tools before they have developed enough investing maturity to justify those costs.

2.2 Limitations of Existing Systems

Existing retail finance platforms often solve only one part of the workflow. Some are good at stock screening but weak in portfolio context. Others offer news or charting but provide poor beginner guidance. Many are highly optimized for one geography only. Several rely on proprietary or paid APIs that make cloning, learning, experimentation, or academic deployment difficult. Some interfaces are also built primarily for experienced users and present metrics without enough explanation, making them intimidating to new investors.

The repository README explicitly states that this project is inspired by well-known market sites such as Screener.in, Tickertape, and Moneycontrol. However, rather than simply reproducing those websites, this project synthesizes their most useful concepts into a local-first educational platform with a more explicit focus on implementation transparency, multi-source data assembly, and explainable decision support.

2.3 Proposed Solution

Stock Metrics proposes a modular web platform that supports market search, quote retrieval, chart history, fundamentals display, news aggregation, document links, user-owned watchlists and portfolios, local note taking, natural-language stock screening, AI insights, educational course discovery, a Q/A workspace backed by local LLM infrastructure, and a personalized recommendation engine. The key engineering idea is to separate concerns clearly: UI components render data, adapters abstract access to data, providers integrate public sources, storage repositories manage browser persistence, and the AI layer operates through consistent provider interfaces. This makes the application easier to extend, test, and reason about.

6. System Architecture and Design

6.1 Architectural Philosophy

The architecture of Stock Metrics is shaped by five guiding principles: modularity, local-first persistence, static-first delivery, graceful degradation, and explainable AI. Instead of a monolithic backend and opaque AI prompts, the system composes focused modules that each solve a well-defined concern. UI components do not call raw providers directly. Data retrieval passes through adapters, route handlers, provider integrations, and cache layers. This separation reduces coupling and makes the system easier to understand, test, and extend.

6.2 High-Level Layers

  1. Presentation Layer: Next.js pages and feature components provide the UI for dashboard, screener, watchlist, portfolio, alerts, agentic analysis, Q/A, learning, and authentication.
  2. Client State and Storage Layer: Zustand stores maintain theme, UI mode, and auth state, while IndexedDB preserves long-lived user records and analysis artifacts.
  3. Client Domain Layer: Adapters aggregate quotes, history, fundamentals, news, documents, FX, and AI responses into app-ready bundles.
  4. Server Boundary Layer: Next.js route handlers proxy free external APIs, normalize responses, and attach cache headers.
  5. Provider Layer: Dedicated provider modules integrate Yahoo Finance, SEC EDGAR, RSS feeds, MFAPI, NSE data, AMFI text, and external utility APIs.

6.4 Core Navigation and Pages

The active navigation bar exposes the following main modules: Dashboard, Screener, Watchlist, Portfolio, Alert, Personalized Agent, Chat with AI, and Learning. Supporting auth-related routes include Login, Register, Forgot Password, and Account. The root page redirects to the dashboard. A dynamic route under /dashboard/[market]/[symbol] renders the stock or mutual fund detail page.

6.5 State and Persistence Design

UI preferences are stored with Zustand persistence under the key stock-metrics-ui, retaining theme, interface mode, default market, and currency preference for U.S. stocks. Authentication state is held in a dedicated store. Long-lived browser data is saved in IndexedDB object stores: users, session, watchlists, portfolioTxns, notes, customScreens, priceAlerts, alertMessages, and kv. This design keeps user-controlled content local by default.

6.6 Caching Strategy

The project uses caching at multiple levels. On the client side, React Query governs request freshness and garbage collection, while a lightweight in-memory TTL map in fetch-cache.ts avoids repeated fetches for the same endpoint. On the server side, route handlers and provider helpers use a memory-based withServerCache function. External requests also use Next.js revalidation where appropriate. This combination balances responsiveness with a realistic expectation that free public data providers should not be called excessively.

Architectural highlight: the system is not “frontend only” and not a traditional heavy backend either. It is best described as a static-first web application with local persistence and targeted serverless proxy routes for live provider normalization.

7. Module-Wise Implementation Details

7.1 Dashboard and Stock Detail Module

The dashboard is the default entry experience. It presents market-aware discovery, hero metrics, coverage indicators, and transitions into detailed company analysis. The detail page assembles a StockDetailBundle by first resolving the search entity and then fetching quote, historical series, fundamentals, news, and documents in parallel. This is a sound design choice because it minimizes latency and aligns each feature panel with a clearly defined data contract.

The stock detail view includes:

  • Company/about information and website links.
  • Quote display, change tracking, and market timing information.
  • Key metrics with missing-value suppression instead of misleading placeholders.
  • Financial statement tables for profit and loss, quarterly results, balance sheet, and cash flow.
  • Consolidated or standalone view support where applicable.
  • Interactive price chart panels with different time windows.
  • AI insight summaries and beginner-friendly checks.
  • Peer suggestions and comparison support.
  • News, document links, and locally stored notes.
  • Excel export for financial statements.
  • USD/INR presentation support for U.S. counters.

7.2 Universal Search Module

Search is a foundational module because almost all user journeys begin with entity discovery. The search provider integrates three distinct public indexes: U.S. listings are derived from Nasdaq Trader symbol directories, Indian equities are derived from the NSE equity list CSV, and Indian mutual funds are resolved via MFAPI or AMFI text fallbacks. Search results are scored using normalized symbol, display symbol, name, and alias matching, with additional bias for market context. The implementation also merges live results with the internal demo universe to ensure continuity even when public index loading fails.

7.3 Screener Module

The screener workbench is one of the heaviest modules in the repository and demonstrates substantial feature engineering. It defines a large screening row model with valuation, profitability, balance sheet, growth, ownership, price/volume, and technical fields. It supports both direct filter manipulation and natural language parsing through the heuristic AI provider. The module also includes preset strategies such as Quality Compounders, Value + Yield, Growth Leaders, Low Leverage, Momentum, and Largecap Stability.

Beyond static filtering, the screener computes or derives several technical values including moving averages, RSI, volatility, drawdown, and relative return measures from historical data. Results are rendered through a virtualized table so the interface remains efficient even when result sets grow.

7.4 Watchlist Module

The watchlist module enables users to create multiple named watchlists, add and remove symbols, and inspect quick visual cues such as sparkline movement and short-term return blocks. Watchlists are stored locally. The module also seeds an initial demo watchlist if no watchlists are found, improving first-run usability. A watchlist row links directly into the detail page of the selected security.

7.5 Portfolio Module

The portfolio module captures buy and sell transactions, derives holdings from transaction history, computes weighted average cost, calculates current value and profit/loss, and normalizes cross-currency positions into INR using the latest available USD to INR rate. Even though the pricing source is delayed and best-effort, the module still provides a coherent personal accounting view. Allocation percentages are visualized with lightweight bars rather than requiring a heavier chart dependency.

7.6 Alerts Module

Price alerts are stored per user and evaluated client-side by a background monitor mounted at the app provider level. The monitor polls approximately every 45 seconds, retrieves latest quotes for enabled alerts, checks condition crossings through a pure evaluation function, writes alert messages to IndexedDB, and optionally invokes a server email route backed by Gmail SMTP credentials. This design is elegant for a low-cost architecture, though it also means alert checks only run while a user session is active in the browser.

7.7 Agentic Personalized Recommendation Module

The personalized agent module is a major differentiator in the codebase. The form collects user demographics, dependents, employment profile, monthly cash flow, taxes, assets, retirement balances, liabilities, emergency fund cover, goals, time horizon, risk preference, liquidity need, and market scope. The engine then:

  1. Normalizes and validates profile input.
  2. Builds a financial profile including surplus, debt burden, net worth, emergency fund gap, current allocation, and ideal allocation.
  3. Resolves the investment universe based on market scope.
  4. Loads security data and risk context.
  5. Computes technical, sentiment, fundamentals, DCF, and fit-based scores.
  6. Applies weighted personalized scoring.
  7. Produces ranked recommendations, action notes, and portfolio suggestions.
  8. Supports PDF export of the final report.

The default scoring weights in the code are 40% stock quality, 25% risk compatibility, 20% portfolio fit, and 15% life-stage fit. This is a noteworthy design because it reflects a multi-factor recommendation process rather than a single black-box confidence number.

7.8 Q/A Module

The Q/A workbench is a general-purpose chat interface that integrates with Ollama. It checks model availability, discovers installed models, falls back from the configured default when needed, and formats assistant responses into headings, lists, paragraphs, and code blocks. Chat history is autosaved into IndexedDB key-value storage so the conversational state persists locally. Unlike the heuristic learning assistant, this route is intended for broader-purpose interaction and depends on a running local Ollama server.

7.9 Learning Module

In the current routed application, the Learning page presents a curated free-course catalog from Khan Academy, MIT OpenCourseWare, and Open Yale Courses. Users can filter by category and search by topic. This makes the learning section practical for project demonstrations because it offers visible educational value without requiring external AI configuration.

At the codebase level, the repository also contains a separate LearningAssistant component and local markdown content files under src/content/learning. These assets are not currently wired to the active Learning route, which suggests an evolution path from a local knowledge-base assistant toward the current curated-course hub. This is valuable evidence of iterative development rather than a purely static one-shot implementation.

7.10 Authentication Module

The auth system uses an adapter pattern. In default mode, a local auth adapter stores registered demo users in IndexedDB and keeps a remembered session in IndexedDB or a temporary session in sessionStorage. Google-based actions intentionally throw informative errors unless Firebase is configured. An optional Firebase adapter provides email/password, Google sign-in, session persistence, password reset, and account deletion handling when the relevant environment variables are present.

8.2 Fallback and Transparency Strategy

One of the strongest engineering decisions in this project is that fallback behavior is explicit rather than hidden. For example, if the quote or historical feed fails, demo history can be generated. If U.S. fundamentals are available only partially, the live SEC bundle is merged with demo statements and peer information. If FX fails, cached data is used and flagged as stale. Missing metrics are hidden instead of guessed. This approach protects the integrity of the user interface by preferring omission over misinformation.

8.3 Heuristic AI Layer

The heuristic AI provider is the default analysis engine. It offers statement summaries, AI insight generation, beginner assessment, peer suggestion, learning question answering, and screener query parsing. These capabilities are implemented through deterministic logic rather than remote generative inference, which makes the platform operational even without paid API access.

The heuristic layer uses the following notable techniques:

  • Metric extraction from structured bundles.
  • Trend phase detection from price returns over time.
  • Volatility and maximum drawdown estimation from historical series.
  • Fraud-risk flags based on leverage, pledge, margin weakness, and sales-profit mismatch.
  • Lexicon-based news sentiment with positive and negative token lists.
  • Beginner guidance translated into simple checks and recommendations.

8.4 Personalized Engine Logic

The personalized engine combines household finance and security analysis. It computes monthly investable surplus, EMI burden, emergency fund shortfall, risk profile, allocation gaps, local portfolio concentration, technical signals, sentiment distributions, risk levels, DCF valuation state, tax holding guidance, and recommendation fit. The DCF logic models projected free cash flows for five years and adds a terminal value to produce an intrinsic value per share when sufficient inputs exist. Personalized scoring blends stock quality, risk compatibility, portfolio fit, and life-stage fit into a final recommendation output such as BUY, HOLD, or SELL.

8.5 Alert Logic

The price alert evaluator is intentionally simple and testable. It checks whether a price is above or below the target and triggers only when the condition changes from unmet to met. This avoids repeated notifications while the price remains on the same side of the threshold. The message builder then formats a readable subject and body for both in-app storage and email delivery.

8.6 Export Utilities

Two export paths exist in the application. First, financial statements can be written to a multi-sheet Excel workbook, including separate consolidated and standalone sheets when available. Second, the personalized agent can generate a PDF report summarizing user profile, household finance score, recommendation, alternatives, and model notes. These export capabilities strengthen the project’s usefulness in real workflows and demonstrate applied data transformation.

10. Deployment Model and Operational Characteristics

10.1 Local Execution

The application is designed for easy local execution. After installing dependencies, the developer can run the application using standard Next.js development mode. This simplicity is a major advantage in an academic environment because it removes heavy setup barriers and allows project evaluation on ordinary machines.

10.2 Static-First Hosting Model

Although the app includes route handlers for provider proxying, its general design remains static-first. Public pages and route structure are friendly to static output patterns, while serverless function-style routes handle dynamic provider operations. This makes the system suitable for Vercel as well as Cloudflare Pages with the build:cloudflare script.

10.3 Environment Variables

The following optional groups are exposed in the environment template:

  • OpenAI-compatible endpoint settings for optional AI answering.
  • Ollama base URL and model selection.
  • Gmail SMTP credentials for alert email delivery.
  • Firebase client configuration for hosted authentication.
  • Feature toggles such as default AI provider and Firebase enablement.

10.4 Operational Notes

  • Q/A functionality depends on a reachable Ollama instance.
  • Email alerts depend on SMTP credentials and a logged-in user email.
  • Firebase auth is optional and disabled by default.
  • Data freshness depends on the quality and availability of free external providers.
  • Client-owned data stays in the browser and can be lost if browser storage is cleared.

11. Security, Privacy, Reliability, and Limitations

11.1 Positive Security and Privacy Characteristics

  • User watchlists, portfolios, notes, custom screens, and alert history are stored locally rather than sent to a central database by default.
  • External integrations are optional, reducing mandatory attack surface.
  • Auth behavior is abstracted, making it possible to swap implementations without rewriting feature modules.
  • Route handlers validate basic request requirements and return bounded error responses.

11.2 Practical Limitations and Risks

  • The local auth adapter uses demo-oriented password hashing and is not suitable for production-grade public deployment.
  • Client-side background alert checks are not always-on monitoring; they require an active browser session.
  • In-memory server cache is process-local and not a shared distributed cache.
  • Free provider coverage for Indian fundamentals is limited and partially supplemented by demo content.
  • News relevance and sentiment are heuristic and should be treated as supportive context, not investment truth.
  • Q/A answers are dependent on local model quality when Ollama is used.
  • Browser storage can be cleared by the user or browser policies, affecting persistence.

11.3 Reliability Strategy

Despite these limitations, the project is robust in a practical sense because it does not depend on a single remote provider or a single AI model. It has a multi-layer fallback strategy, explicit stale labeling, demo data continuity, local storage of user work, and modular adapters that can be replaced in future versions. For a student project, this is a strong reliability posture because it reflects awareness of operational failure modes rather than assuming ideal conditions.

  • Promote the unused local learning-assistant knowledge-base path into the active Learning route or offer both modes.
  • Introduce server-side scheduled alert checking for true 24x7 notifications.
  • Add benchmark-relative analytics such as beta versus market indexes and sector ETFs.
  • Improve portfolio analytics with realized gains, sector trends, and tax lots.
  • Expand AI explainability with richer score breakdown visualizations.
  • Add stronger production security for auth and secrets management.
  • Extend automated testing to end-to-end coverage.
  • Provide export to Word/PDF from within the application itself.

13. Conclusion

Stock Metrics is a well-structured, ambitious, and technically substantial finance platform. It addresses a real user problem, employs a modern web stack, integrates multiple public data sources, and remains usable even when premium services are unavailable. Its architecture demonstrates thoughtful separation of concerns, especially in the data adapter/provider model and the combination of local-first persistence with lightweight server boundaries.

What makes the project especially strong as a final year submission is not only the range of features but the maturity of the engineering decisions behind them. The system is transparent about data quality, intentionally conservative about missing information, test-backed in key logic areas, and extensible in ways that leave clear room for future research and product enhancement. In summary, the project is a strong example of applied full-stack software engineering in the financial domain.
